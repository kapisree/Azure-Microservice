# Known Limitations — v0.1.0

## Security findings carried forward (from `docs/security/0.1.0-disposition.md`)

### SEC-001 — accepted (expires 2026-12-31)
Unauthenticated read access to synthetic claim data. Accepted under ADR-004 as the spec's deliberately chosen v1 boundary. Revisit before any real claimant data is substituted in, or before this expiry, whichever is sooner.

### SEC-002 — accepted, low (no expiry required)
No rate limiting in v1, out of scope per the spec's Won't Have. Kubernetes resource requests/limits bound the DoS blast radius to the pod's own allocation.

### SEC-003 — deferred (expires 2026-09-30)
Base container image is not digest-pinned (`mcr.microsoft.com/...:tag` rather than `@sha256:...`). Deferred because resolving the current manifest digest requires a live registry, not available in the original offline review session.

### SEC-004 — deferred (expires 2026-09-30)
The CI vulnerability scan step (`dotnet list package --vulnerable`) does not fail the build on findings — it always exits 0. Deferred pending a product decision on which severities should gate the build.

### SEC-005 — deferred (expires 2026-09-30)
The AKS API server has no `aadProfile`/`authorizedIPRanges` configured, so the control plane is publicly reachable with only Kubernetes RBAC as a boundary. Deferred pending a decision on a stable authorized IP range or Azure AD group mapping.

## Outstanding second-reviewer confirmation

Per `art-threat-driven-security`, the disposition's severities are solo-author
provisional. The verifier findings posted on PR #6 flag two confirmations
still outstanding for a second reviewer: (1) whether decomposing the original
"high"-rated public LoadBalancer exposure (per ADR-004 and the fresh-eyes
review) into SEC-001 (medium) + SEC-002 (low) is sound rather than
severity-splitting, and (2) whether SEC-001's `VC:L` CVSS input (synthetic
data today) should hold given the threat model notes the data-synthetic
property is a documentation promise, not a code-enforced control. RELEASE
proceeds on the existing provisional scoring; re-score and re-run RELEASE's
gate check if a second reviewer re-rates either as high.

GPG secret key available (`gpg --clearsign` failed with "no default secret
key"). Follow-up: once GPG is configured for this repo, re-tag with
`git tag -d v0.1.0 && git tag -s v0.1.0 -m "Release v0.1.0" <commit> && git push origin v0.1.0 --force`.
