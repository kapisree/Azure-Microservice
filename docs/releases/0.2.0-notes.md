# v0.2.0 Release Notes

## Added
- API-key authentication on `GET /claims` and `GET /claims/{claimId}`: requests must present a valid `X-Api-Key` header matching the configured key, or receive a uniform `401` (REQ-309, REQ-311). `GET /health` remains unauthenticated (REQ-310).
- `ApiKeyFilter`/`Authorize` kernel module (ADR-005), wired via `app.MapGroup("/claims").AddEndpointFilter<ApiKeyFilter>()`.
- Dafny proof (`verification/claims_api_auth/api_key_filter.dfy`) establishing auth takes precedence over route-level validation (a missing/wrong key with an invalid `claimId` always returns `Unauthorized`, never `BadRequest`) and that an empty configured key can never authorize any presented value (REQ-309).
- Kubernetes `Secret` (`infra/k8s/secret.yaml`) and `Deployment` env wiring (REQ-312) sourcing `ApiKey:Value` from the cluster Secret rather than a static manifest value.
- CI workflow extension (REQ-313): fail-fast guard on the `CLAIMS_API_KEY` repo secret, namespace-first apply order, temp-file-based Secret creation/update (never via argv), then manifest apply and a rollout restart so running pods pick up the current key.
- `appsettings.Development.json` placeholder key for local development (REQ-314).

## Changed
- `Program.cs` refactored to route both claims endpoints through a `MapGroup("/claims")` so the filter applies uniformly.
- `.github/workflows/claims-api.yml`'s deploy job extended with the Secret/namespace/rollout steps above; the existing build/test/scan job is unchanged.

## Fixed
- **SEC-006** (high, fixed before RELEASE): `ApiKeyFilter.Authorize` treated an empty configured key and an empty presented key as a match (`CryptographicOperations.FixedTimeEquals` defines two zero-length spans as equal), fail-opening authentication whenever `ApiKey:Value` resolved to empty — the `appsettings.json` production default. Fixed via the `impl/sec-sec-006` back-edge (PR #13): `Authorize` now rejects whenever either side is null/empty.

## Security
SECURITY review at `docs/security/0.2.0-review.md`, disposition at `docs/security/0.2.0-disposition.md`. 2 new findings raised, plus 4 carried forward unchanged from v0.1.0 and 1 superseded. At release time: 0 critical, 0 high open, 4 medium accepted/deferred, 1 low. SECURITY exit gate (`art-threat-driven-security`) satisfied — the one high finding (SEC-006) was fixed via the back-edge protocol before this disposition was written, not accepted or deferred.

- **Fixed:** SEC-006 (fail-open empty-key authentication bypass — back-edge `impl/sec-sec-006`, PR #13).
- **Superseded:** SEC-001 (the v0.1.0 review's "fully unauthenticated `/claims` access" finding no longer describes the system now that a valid API key is required) → replaced by SEC-007.
- **Accepted:** SEC-007 (residual plaintext-channel API-key capture risk over `tb-1`, narrowed not closed — TLS still absent; expires 2026-12-31), SEC-002 (no rate limiting, carried forward unchanged, low).
- **Deferred (unchanged from v0.1.0, this feature doesn't touch the affected files):** SEC-003 (base image not digest-pinned), SEC-004 (vulnerability scan step doesn't fail the build), SEC-005 (AKS API server has no `aadProfile`/authorized IP ranges) — all expire 2026-09-30.

A second-reviewer confirmation of SEC-007's provisional severity (specifically the `AV:N` CVSS vector choice) is recorded as outstanding in the verifier findings on PR #14/PR #15 — see `docs/security/0.2.0-disposition.md`'s header. SEC-006 is `fixed`, not provisional — it is immune to that confirmation outcome.

## Breaking
- Any existing caller of `GET /claims` or `GET /claims/{claimId}` without an `X-Api-Key` header now receives `401 Unauthorized` instead of `200 OK`. `GET /health` is unaffected.
