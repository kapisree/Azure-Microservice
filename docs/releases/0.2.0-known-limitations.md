# Known Limitations — v0.2.0

## Security findings carried forward (from `docs/security/0.2.0-disposition.md`)

### SEC-006 — fixed (no expiry, closed)
`ApiKeyFilter.Authorize` fail-opened (returned authorized) when both the
configured key and the presented key were empty strings, because
`CryptographicOperations.FixedTimeEquals` treats two zero-length spans as
equal. Fixed via the `impl/sec-sec-006` back-edge (PR #13) before this
disposition was written — `Authorize` now rejects whenever either side is
null/empty. Not carried forward as an open risk.

### SEC-007 — fixed 2026-06-23 via `impl/sec-sec-007` (PR #20), supersedes SEC-001
**Closed 2026-06-23.** At the time v0.2.0 was tagged, SEC-007 carried the
author's provisional `accepted` disposition described below. The
second-reviewer confirmation that disposition was explicitly contingent
on (see "Outstanding second-reviewer confirmation" below) landed: an
independent subagent review recomputed the CVSS v4.0 score from ~5.1
medium to ~7.7 high (disagreement on the `VC` impact metric — see
`docs/security/0.2.0-disposition.md`'s SEC-007 entry for the full vector
comparison). Per `art-threat-driven-security`, high findings cannot be
`accepted`; SEC-007's status changed from `accepted` to `open`, and the
back-edge protocol activated on `impl/sec-sec-007`. The fix below landed,
PR #20 merged (`db1af6f`), VALIDATE re-ran clean on `main`, and the
disposition now records `status: fixed`. This v0.2.0 tag itself is not
retroactively re-released — this note documents that the risk it
originally accepted is now closed, not that v0.2.0 itself changed.

**Fix landed on `impl/sec-sec-007` (PR #20):** the Service is now
`ClusterIP` (no public plaintext LoadBalancer) fronted by an nginx
`Ingress` (`infra/k8s/ingress.yaml`) terminating TLS via a cert-manager
`ClusterIssuer` (`infra/k8s/cluster-issuer.yaml`) for
`claims-api.aicodeingdemo.com`, Let's Encrypt production, HTTP-01
challenge. Two prerequisites were **not** provisioned by this repo's IaC
and required manual cluster-operator action before the cert could
actually issue: (1) installing `ingress-nginx` and `cert-manager` on the
AKS cluster (`infra/bicep/main.bicep` only stands up ACR + the AKS
cluster itself), and (2) pointing `claims-api.aicodeingdemo.com`'s DNS A
record at the `ingress-nginx` controller's LoadBalancer IP. Both were
confirmed complete on the live cluster before this disposition was
closed to `fixed` — per the PR #20 verifier's caution against declaring
the risk closed on merged manifests alone.

Original text (2026-06-22, kept for the record): the API-key channel
(`tb-1`) is still plaintext HTTP, no TLS, exposed via a public
`LoadBalancer` with no source-IP restriction. An attacker positioned to
observe traffic (on-path/MITM) can capture a valid `X-Api-Key` value and
replay it. Accepted because TLS termination is out of scope for this
offline-reviewed product per ADR-004's amendment; this acceptance is
conditional on claim data remaining synthetic (ADR-004's own "not
code-enforced, conditional and revocable" caveat) — revisit before any
real claimant data is substituted in, or before this expiry, whichever is
sooner. The provisional `AV:N` CVSS vector choice underlying this
acceptance is recorded as awaiting second-reviewer confirmation (see
below).

### SEC-001 — superseded by SEC-007

`/claims/{claimId}` now require a valid API key. SEC-001's original
wording no longer matches the system; its residual risk is carried

resource requests/limits bound the DoS blast radius to the pod's own

**Resolved 2026-06-23 — see the SEC-007 follow-up note above.** Per
`art-threat-driven-security`, the disposition's severities for SEC-006 and
SEC-007 were solo-author provisional. SEC-006 is `fixed` regardless of how
its score is finally confirmed, so this confirmation did not affect that

## Disposition status convention note

**Resolved 2026-06-23 (verifier finding #1, PR #20).** `docs/security/
0.2.0-disposition.md` uses `status: superseded` for SEC-001. At v0.2.0's
original writing this was ad hoc, not yet enumerated by
`art-threat-driven-security`. It has since been formally enumerated
(retro `2026-06-22-sec-006-process-gap` proposal #1, applied in PR #19) —
see `constitution.md`'s `art-threat-driven-security` and
`docs/security/RUBRIC.md` for the current rule, including the guard
requiring a `status: fixed` successor or second-reviewer confirmation
before `superseded` may apply to a critical/high original. No further
retrospective proposal is needed; this note is left in place only as the
historical record of the gap it once flagged.

## Dockerfile build-context fix (post-v0.2.0, not a security finding)

**2026-06-24, PR #24.** After v0.2.0 was tagged, the Python→.NET tooling
conversion (PR #23) added `DemoGreeting`/`RenderDashboard`/`RenderDoc` to
`ClaimsApi.sln`. `src/ClaimsApi/Dockerfile` ran `dotnet restore
ClaimsApi.sln` against the whole solution but only `COPY`'d in the
`ClaimsApi`/`ClaimsApi.Tests` csproj files, so `docker build` broke —
restore failed looking for projects not present in the image build
context. Fixed by restoring `src/ClaimsApi/ClaimsApi.csproj` directly
instead of the `.sln`, which also decouples the container build from
solution-wide tooling projects it never needed. Verified locally:
`docker build` succeeds, and the running container answers `/health`,
rejects `/claims` without an API key (401), and serves the seeded claims
list with the configured key.

This is a build-mechanics regression, not a vulnerability, and does not
change SEC-003's "Dockerfile was not touched" note above as of v0.2.0 —
the base image lines (`mcr.microsoft.com/dotnet/sdk:8.0` /
`mcr.microsoft.com/dotnet/aspnet:8.0`, still not digest-pinned) were not
touched by this fix either. This v0.2.0 tag itself is not retroactively
re-released; this note documents a post-tag fix to the same file SEC-003
describes, for traceability.
