# Known Limitations ŌĆö v0.2.0

## Security findings carried forward (from `docs/security/0.2.0-disposition.md`)

### SEC-006 ŌĆö fixed (no expiry, closed)
`ApiKeyFilter.Authorize` fail-opened (returned authorized) when both the
configured key and the presented key were empty strings, because
`CryptographicOperations.FixedTimeEquals` treats two zero-length spans as
equal. Fixed via the `impl/sec-sec-006` back-edge (PR #13) before this
disposition was written ŌĆö `Authorize` now rejects whenever either side is
null/empty. Not carried forward as an open risk.

### SEC-007 ŌĆö RESCORED 2026-06-23 to `severity: high, status: open`, supersedes SEC-001
**Follow-up note (2026-06-23): this entry is stale.** At the time v0.2.0
was tagged, SEC-007 carried the author's provisional `accepted` disposition
described below. The second-reviewer confirmation that disposition was
explicitly contingent on (see "Outstanding second-reviewer confirmation"
below) has since landed: an independent subagent review recomputed the
CVSS v4.0 score from ~5.1 medium to ~7.7 high (disagreement on the `VC`
impact metric ŌĆö see `docs/security/0.2.0-disposition.md`'s SEC-007 entry
for the full vector comparison). Per `art-threat-driven-security`, high
findings cannot be `accepted`; SEC-007's status changed from `accepted` to
`open`, and the back-edge protocol is in effect on `impl/sec-sec-007`. This
v0.2.0 tag itself is not retroactively un-released, but the limitation
below should be read as superseded by that branch's eventual `status:
fixed` disposition, not as the closing word on this risk.

**Fix landed on `impl/sec-sec-007`:** the Service is now `ClusterIP` (no
public plaintext LoadBalancer) fronted by an nginx `Ingress`
(`infra/k8s/ingress.yaml`) terminating TLS via a cert-manager
`ClusterIssuer` (`infra/k8s/cluster-issuer.yaml`) for
`claims-api.aicodeingdemo.com`, Let's Encrypt production, HTTP-01
challenge. Two prerequisites are **not** provisioned by this repo's IaC
and remain manual cluster-operator steps before the cert will actually
issue: (1) installing `ingress-nginx` and `cert-manager` on the AKS
cluster (`infra/bicep/main.bicep` only stands up ACR + the AKS cluster
itself), and (2) pointing `claims-api.aicodeingdemo.com`'s DNS A record at
the `ingress-nginx` controller's LoadBalancer IP once it exists. Until
both are done, the Ingress will exist but the ACME HTTP-01 challenge
cannot complete.

Original text (2026-06-22, kept for the record): the API-key channel
(`tb-1`) is still plaintext HTTP, no TLS, exposed via a public
`LoadBalancer` with no source-IP restriction. An attacker positioned to
observe traffic (on-path/MITM) can capture a valid `X-Api-Key` value and
replay it. Accepted because TLS termination is out of scope for this
offline-reviewed product per ADR-004's amendment; this acceptance is
conditional on claim data remaining synthetic (ADR-004's own "not
code-enforced, conditional and revocable" caveat) ŌĆö revisit before any
real claimant data is substituted in, or before this expiry, whichever is
sooner. The provisional `AV:N` CVSS vector choice underlying this
acceptance is recorded as awaiting second-reviewer confirmation (see
below).

### SEC-001 ŌĆö superseded by SEC-007

resource requests/limits bound the DoS blast radius to the pod's own
allocation. Unaffected by this release.

## Outstanding second-reviewer confirmation

**Resolved 2026-06-23 ŌĆö see the SEC-007 follow-up note above.** Per
`art-threat-driven-security`, the disposition's severities for SEC-006 and
SEC-007 were solo-author provisional. SEC-006 is `fixed` regardless of how
its score is finally confirmed, so this confirmation did not affect that
finding's status. SEC-007's `accepted` disposition depended on the second
reviewer confirming `docs/security/0.2.0-review.md`'s `AV:N` CVSS vector
choice (network, not adjacent, given the public `LoadBalancer` exposure)
and the score staying at or below medium (~5.1). The verifier findings
posted on PR #14 and PR #15 flagged this explicitly. The second reviewer
(independent subagent dispatch) instead rescored SEC-007 to high (~7.7) ŌĆö
the disposition has changed to a back-edge candidate (`impl/sec-sec-007`),
per the note above.

## Disposition status convention note

`docs/security/0.2.0-disposition.md` uses `status: superseded` for SEC-001
ŌĆö a value not enumerated by `art-threat-driven-security` (`open`/`fixed`/
`accepted`/`deferred`). It is excluded from cap accounting and does not
represent an open risk. A `/retrospective` proposal to formalize this
status (or an equivalent path for findings closed by supersession) is
recommended before it is reused in a future release.
