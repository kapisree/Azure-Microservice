# Migration to v0.2.0

## Breaking change

`GET /claims` and `GET /claims/{claimId}` now require a valid `X-Api-Key`
header. Requests without one, or with a non-matching value, receive `401
Unauthorized` instead of the previous `200 OK`. `GET /health` is
unaffected — no client of that endpoint needs to change.

## Steps

1. **Before deploying**, set the `CLAIMS_API_KEY` repository secret (GitHub
   Actions → Settings → Secrets) to the value you want the deployed API to
   require. The CI workflow (`.github/workflows/claims-api.yml`) now fails
   fast if this secret is unset or empty — it will not silently deploy
   with an empty key.
2. Deploy as before (push to `main`, or merge a PR). The extended deploy
   job applies the namespace first, then creates/updates the
   `claims-api-key` Kubernetes `Secret` from `CLAIMS_API_KEY` (via a
   temp file, never via argv), applies the remaining manifests, and
   restarts the rollout so already-running pods pick up the current key.
3. **Update every existing caller** of `GET /claims`/`GET /claims/{claimId}`
   to send `X-Api-Key: <the value you set in step 1>` on every request.
   Callers that only use `GET /health` need no change.
4. For local development, `appsettings.Development.json` ships with a
   placeholder key (`local-dev-placeholder-key`) — use that value when
   calling the API locally, or override `ApiKey:Value` in your own local
   config.

## Rollback

Redeploying a pre-0.2.0 image removes the `ApiKeyFilter` requirement
entirely (the route group and filter did not exist before this release).
The `claims-api-key` Secret and `CLAIMS_API_KEY` repository secret can be
left in place — they are simply unused by a pre-0.2.0 deployment.
