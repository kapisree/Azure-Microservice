## Findings

### SEC-006 (fixed before this document was written)
```yaml
---
id: SEC-006
severity: high
cite: [dfe-6, CWE-697, "OWASP API2:2023"]
affects: [src/ClaimsApi/ApiKeyFilter.cs:21, src/ClaimsApi/appsettings.json:9]
status: fixed
fix_pr: 13
fix_commit: c9ec7d4
---
```
`ApiKeyFilter.Authorize(string? presented, string configured)` null-checked
`presented` but never checked either argument for being an *empty* string
before calling `CryptographicOperations.FixedTimeEquals`. `FixedTimeEquals`
defines two zero-length spans as equal, so `Authorize("", "")` returned
`true`. `appsettings.json`'s production default for `ApiKey:Value` is
`""` (Task A2's deliberate placeholder, overwritten only by the CI deploy
job's Secret injection — see SEC-004/SEC-005's existing notes on the
deploy pipeline's reliance on operational discipline rather than
fail-closed defaults). Any deployment where the `CLAIMS_API_KEY` Secret
injection has not yet run (first apply before the "Create/update API key
Secret" CI step, a rollback, or a misconfigured environment) would resolve
`configured` to `""`, and any caller presenting an empty (but
header-present) `X-Api-Key` value would be treated as authenticated —
a full, silent bypass of `dfe-6`'s gate granting unauthenticated callers
the exact read access ADR-004's amendment intended to require a key for.

CVSS v4.0 base ~7.1 (AV:N/AC:L/AT:P/PR:N/UI:N/VC:H/VI:N/VA:N) — network
reachable, trivial to trigger (an empty header value), full confidentiality
impact on all claims data once the precondition (empty configured key)
holds; `AT:P` reflects that the precondition is a misconfiguration/timing
state rather than always-true, consistent with this rubric's own worked
example contrasting `AT:P` cases. Per `art-threat-driven-security`, high
severity cannot be accepted or deferred — this triggered the back-edge
protocol: `impl/sec-sec-006` branched from `main`, a failing unit test
(`Authorize_WithEmptyPresentedAndEmptyConfigured_MustNotAuthorize`)
confirmed the bypass before any fix code was written (`art-test-first`),
`Authorize` was changed to reject whenever either `presented` or
`configured` is null/empty via `string.IsNullOrEmpty`, the Dafny model in
`verification/claims_api_auth/api_key_filter.dfy` was updated to require
both strings non-empty plus a lemma proving an empty configured key can
never authorize any presented value, PR #13 merged to `main`, and VALIDATE
was re-run clean. Status: **fixed**.

### SEC-007 (supersedes SEC-001)
```yaml
---
id: SEC-007
severity: medium
cite: [tb-1, dfe-6, CWE-319, "OWASP API2:2023"]
affects: [src/ClaimsApi/Program.cs, infra/bicep/main.bicep]
supersedes: SEC-001
---
```
SEC-001 (`docs/security/0.1.0-disposition.md`, `status: accepted`) recorded
`tb-1` as fully unauthenticated read access to `/claims`. This feature
closes that gap — both routes now require a valid `X-Api-Key` (`dfe-6`,
SEC-006 fixed above) — so SEC-001 as originally worded no longer describes
the current system and would mislead a future reader if left untouched
(per the auth spec's Open Question #4). The residual risk SEC-001 partially
named is renamed and re-scored here: `tb-1` is **narrowed, not closed**
(per ADR-004's amendment and the threat model's own "Trust boundary
tb-1 — narrowed, not closed" section) because the channel is still
plaintext HTTP, no TLS. An attacker positioned to observe `tb-1` traffic
(network MITM, compromised intermediate proxy) can capture a valid
`X-Api-Key` value in transit and replay it as their own — the threat
model's `dfe-6` Tampering finding. CVSS v4.0 base ~4.8
(AV:A/AC:H/AT:P/PR:N/UI:N/VC:L/VI:N/VA:N) — requires a privileged network
position (adjacent, not remote) plus a legitimate caller's traffic to
observe, materially harder to exploit than SEC-001's original
fully-open-internet framing, hence medium rather than SEC-001's prior
medium-for-a-different-reason. Remediation (same as SEC-001's original,
carried forward): TLS termination in front of `claims-api`, deferred to a
future spec — synthetic claim data and the absence of a live ingress in
this offline-reviewed product make this lower urgency than shipping the
auth feature itself.

## Open Questions resolved

- Spec Open Question #3 (rotation-gap disposition, REQ-313(d)): the
  single-key forced-restart rotation gap is sufficiently covered by the
  spec's existing Non-Functional Requirements caveat and does not surface
  as an independent exploitable risk distinct from SEC-007's existing
  `tb-1`/key-capture framing — a captured key is equally exploitable
  whether rotation is manual or automatic, and the rotation mechanism
  itself (CI-driven `kubectl create secret ... --dry-run=client | kubectl
  apply -f -`, followed by an explicit `kubectl rollout restart`) was
  reviewed and introduces no new finding: the temp file is written to
  `mktemp`'s default per-process location and removed (`rm -f`) immediately
  after the apply, never passed via argv, consistent with the threat
  model's named `dfe-6` mitigation. No new SEC-NNN raised for this.
- Spec Open Question #4 (SEC-001 reconciliation): resolved above via
  SEC-007, which supersedes SEC-001.
