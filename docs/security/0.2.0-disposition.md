# Security Disposition — v0.2.0 (Claims Status API authentication, REQ-309..314)

Solo-team provisional severities (per `art-threat-driven-security`): the
author (this session) recorded `severity` provisionally for SEC-006 and
SEC-007. They require confirmation or downgrade by a second reviewer;
until then treat severities as provisional, not final. SEC-002–005 retain
their existing provisional status from `docs/security/0.1.0-disposition.md`,
unchanged. SEC-006 is `fixed` regardless of how its score is finally
confirmed, so the gate is robust to an upgrade there; the one open
sensitivity is SEC-007 — its `accepted` disposition assumes the second
reviewer confirms `docs/security/0.2.0-review.md`'s CVSS recompute (`AV:N`,
medium band) rather than a score crossing into high, which would instead
require treating it as a back-edge candidate.

`status: superseded` (used below for SEC-001) is not one of the statuses
`art-threat-driven-security` enumerates (`open`/`fixed`/`accepted`/
`deferred`). It is used here ad hoc, deliberately: SEC-001 is not open,
not fixed by code, and neither accepted nor deferred in its own right —
it has been replaced by a differently-scoped finding (SEC-007). It carries
no `expiry:` and is explicitly excluded from the cap accounting in the
Gate check below, since it is not itself an open risk. If this pattern
recurs, it should be formalized in `art-threat-driven-security`/
`RUBRIC.md` via `/retrospective` rather than re-invented per release.

## Gate check

- Critical, open: **0**
- High, open: **0** (SEC-006 was high; status is `fixed`, not open)
- Medium, accepted/deferred: **4** (SEC-007, SEC-003, SEC-004, SEC-005) — under the default cap of 5
- Low: **1** (SEC-002) — does not count toward the cap
- Superseded (non-blocking, retained for traceability): **1** (SEC-001)

Per `art-threat-driven-security`, no critical or high finding is **open**:
SEC-006's back-edge completed before this disposition was written
(`impl/sec-sec-006`, PR #13, merged, VALIDATE re-run green) and is
terminally `fixed`, immune to any future rescore. The clean-gate
conclusion as a whole is **contingent, not final**, on one outstanding
item: SEC-007's `accepted` disposition rests on an author-provisional
medium severity that no second reviewer has yet confirmed (no second
reviewer's handle or rationale is recorded in any `rationale:` field
above). If that confirmation instead rescores SEC-007 to high, its
disposition must change from `accepted` to a back-edge candidate before
RELEASE may proceed. RELEASE may proceed once a human reviewer records
that confirmation.

Follow-up: the `status: superseded` value used for SEC-001 above is not
enumerated by `art-threat-driven-security` (`open`/`fixed`/`accepted`/
`deferred`) and has now been used once. Per this PR's verifier review, a
`/retrospective` proposal to formalize it (or an equivalent disposition
path for findings closed by supersession rather than by fix/accept/defer)
should be opened rather than left as a standing "if this recurs" deferral.
