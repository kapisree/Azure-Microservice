### SEC-007 (supersedes SEC-001) — RESCORED 2026-06-23, back-edge candidate
```yaml
---
id: SEC-007
status: open
severity: high
owner: kapisree
second_reviewer: independent-review-agent (subagent dispatch, 2026-06-23)
rationale: "Supersedes docs/security/0.1.0-disposition.md's SEC-001, which described tb-1 as fully unauthenticated and no longer matches the system now that dfe-6 requires a valid API key (SEC-006 fixed). Residual risk: tb-1 remains plaintext HTTP over a public LoadBalancer (infra/k8s/service.yaml type: LoadBalancer, no source-IP restriction), so an on-path attacker who captures a valid key in transit can replay it for full read access to claims data. ORIGINAL author score (docs/security/0.2.0-review.md): CVSS v4.0 ~5.1 medium, AV:N/AC:H/AT:P/PR:N/UI:N/VC:L/VI:N/VA:N — VC:L reasoned the risk was 'narrowed, not closed' by adding auth. SECOND REVIEWER (independent subagent, blind to the original score, re-deriving from infra/k8s/service.yaml + threat-model + ApiKeyFilter.cs/Program.cs directly): CVSS v4.0 ~7.7 high, AV:N/AC:L/AT:P/PR:N/UI:N/VC:H/VI:N/VA:N. The disagreement is VC: a successfully replayed key grants full read access to all claims data via GET /claims and GET /claims/{claimId}, identical to a legitimate caller — CVSS impact is conditioned on successful exploitation, not discounted by how much a compensating control (the API key) narrows the likelihood of getting there. The original VC:L conflated 'narrowed likelihood vs. pre-auth' (a likelihood/AT framing) with confidentiality impact (a severity-component framing). The second reviewer's VC:H is the technically correct application of CVSS v4.0 semantics. AC also moved from H to L: passive sniffing of an unencrypted channel requires no special conditions once positioned, independent of likelihood-of-positioning (captured separately by AT:P, on which both reviewers agree). CONFIRMED SEVERITY: high. Per art-threat-driven-security, high findings cannot be accepted or deferred — this finding's status changes from accepted to open, and the back-edge protocol triggers (impl/sec-sec-007). ADR-004's amendment anticipated exactly this outcome: its 'accept for v1' stance is 'only honorable if SECURITY scores the exposure <= medium' and explicitly cedes to the back-edge protocol otherwise — no ADR re-decision needed, the ADR's own text dictates the fix becomes binding over its prior acceptance."
---
```
