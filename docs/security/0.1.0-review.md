# Security Review — v0.1.0 (Claims Status API)

## Summary

5 findings (0 critical, 0 high, 4 medium, 1 low). Reviewed against
`docs/architecture/2026-06-20-claims-status-api-threat-model.md` and the
merged implementation on `main` (PR #5: `src/ClaimsApi/`, `infra/`,
`.github/workflows/claims-api.yml`). No critical or high finding exists;
the SECURITY exit gate (`art-threat-driven-security`) is satisfied without
invoking the `impl/sec-<finding-id>` back-edge.

Two of the five findings (SEC-001, SEC-002) restate risks the threat model
already named and ADR-004 already accepted for v1 — they are recorded here
per the threat model's own instruction ("SECURITY phase findings touching
this exposure must cite `tb-1`, `dfe-1`, or `dfe-2`") rather than as new
discoveries. The remaining three (SEC-003–005) are new findings surfaced by
reviewing the actual implementation artifacts, not anticipated in the
threat model's STRIDE coverage.

## Tools run
- Manual review of `src/ClaimsApi/Program.cs`, `InMemoryClaimsRepository.cs`,
  `Dockerfile`, `infra/bicep/main.bicep`, `infra/k8s/*.yaml`,

## Findings

### SEC-001
```yaml
---
id: SEC-001
severity: medium
cite: [tb-1, dfe-1, dfe-2, CWE-862, "OWASP API1:2023"]
affects: [src/ClaimsApi/Program.cs:9, src/ClaimsApi/Program.cs:12]
---
```
`GET /claims` and `GET /claims/{claimId}` are fully unauthenticated. Any
caller can enumerate individual claim ids (`dfe-1`) or retrieve the entire
seeded dataset in one request (`dfe-2`) with no authorization check. This
is the threat model's named headline risk (`tb-1`), already accepted under
ADR-004 for v1 conditional on the data remaining synthetic. CVSS v4.0 base
~5.3 (AV:N/AC:L/AT:N/PR:N/UI:N/VC:L/VI:N/VA:N) — confidentiality impact
capped at "low" because the seeded data is synthetic by design, not real
claimant data. Remediation (deferred to a future spec per ADR-004):
authentication/authorization on both routes before any real claimant data
is substituted in.

### SEC-002
```yaml
---
id: SEC-002
severity: low
cite: [tb-1, dfe-1, dfe-2, dfe-3, CWE-770, "OWASP API4:2023"]
affects: [src/ClaimsApi/Program.cs:9, src/ClaimsApi/Program.cs:7, src/ClaimsApi/Program.cs:12]
---
```
No endpoint enforces rate limiting or request throttling, so a scripted
caller can generate sustained load against `/claims`, `/claims/{claimId}`,
or `/health` (`dfe-1`/`dfe-2`/`dfe-3`). Severity is low rather than
medium because Kubernetes `resources.requests`/`resources.limits`
(`infra/k8s/deployment.yaml:30-35`) already bound the blast radius to the
pod's own CPU/memory allocation, as the threat model itself notes. CVSS
v4.0 base ~3.1 (AV:N/AC:L/AT:N/PR:N/UI:N/VC:N/VI:N/VA:L). Out of scope per
the spec's Won't Have; accepted under ADR-004.

### SEC-003
```yaml
---
id: SEC-003
severity: medium
cite: [tb-3, dfe-5, CWE-1104, "OWASP A06:2021"]
affects: [src/ClaimsApi/Dockerfile:1, src/ClaimsApi/Dockerfile:10]
---
```
Both Dockerfile stages reference base images by mutable tag
(`mcr.microsoft.com/dotnet/sdk:8.0`, `mcr.microsoft.com/dotnet/aspnet:8.0`)
rather than by content digest. The tag `8.0` is republished by Microsoft
over time (patch updates), so a rebuild on a later date can silently pull a
different image than a prior build — undermining the build reproducibility
and image-integrity property the threat model flags at `tb-3`/`dfe-5`
("the boundary where the container-image asset's Tampering risk actually
lives"). This is not an active compromise, but it removes the ability to
verify two builds used byte-identical base layers. CVSS v4.0 base ~4.8
(AV:N/AC:H/AT:P/PR:N/UI:N/VC:L/VI:L/VA:L). Remediation: pin both `FROM`
lines to a `sha256:` digest, refreshed on a deliberate schedule rather than
implicitly on every build.

### SEC-004
```yaml
---
id: SEC-004
severity: medium
cite: [CWE-1104, "OWASP A06:2021"]
affects: [.github/workflows/claims-api.yml:22]
---
```
The CI workflow's "Vulnerable package scan" step
(`dotnet list ClaimsApi.sln package --vulnerable --include-transitive`)
runs unconditionally but its output is never checked — the step always
exits 0 regardless of whether vulnerable packages are found, so a build
with known-vulnerable transitive dependencies still reports green. CVSS
v4.0 base ~5.1 (AV:N/AC:L/AT:N/PR:N/UI:N/VC:L/VI:L/VA:N) — reflects that a
detectable, known vulnerability class could ship without anyone noticing
the scan output. Remediation: gate the job on the scan's exit behavior
(e.g. `dotnet list package --vulnerable` returns non-zero packages found
text that must be parsed and asserted, since the command itself doesn't
fail the process) or adopt a tool that fails the build natively (e.g.
`dotnet-outdated`, GitHub's Dependabot alerts as a release gate).

### SEC-005
```yaml
---
id: SEC-005
severity: medium
cite: [CWE-284, "OWASP Kubernetes Top 10 K01"]
affects: [infra/bicep/main.bicep:24, infra/bicep/main.bicep:32]
---
```
The AKS cluster (`main.bicep:24`) enables Kubernetes RBAC
(`enableRBAC: true`, line 32) but configures no `aadProfile` (Azure AD
integration) and no `apiServerAccessProfile.authorizedIPRanges`, so the
cluster API server is reachable from any IP and authorization relies
solely on whatever kubeconfig `az aks get-credentials` produces — not on
Azure AD identity. This is not named as an explicit trust boundary in the
current threat model (which scopes `tb-3` to the AKS↔ACR image-pull path
only), but is a real control-plane exposure surfaced by reviewing the
actual Bicep template. CVSS v4.0 base ~5.9 (AV:N/AC:H/AT:P/PR:N/UI:N/VC:L/
VI:L/VA:L) — requires possession of a valid kubeconfig to exploit, which
is itself gated by Azure RBAC on the resource group, but the lack of
network-level restriction widens the attack surface for credential
leakage scenarios. Remediation: add `aadProfile` with
`enableAzureRBAC: true` and/or `apiServerAccessProfile.authorizedIPRanges`
scoped to known CI/operator egress ranges.
